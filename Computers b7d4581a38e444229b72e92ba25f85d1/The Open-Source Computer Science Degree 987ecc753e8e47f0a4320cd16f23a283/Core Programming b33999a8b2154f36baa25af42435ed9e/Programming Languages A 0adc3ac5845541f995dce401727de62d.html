Week 1

Week 2

Key Concepts:

  • The pieces of a Programming Language
    1. Syntax (how to write stuff)
    1. Semantics (what do language features mean)
    1. Idioms (common approaches to do stuff in a language)
    1. Libraries (what functions etc are already written for you?)
    1. Tools (compilers, read-eval-print loops, debuggers etc)
  • Expressions and Variable bindings

    Each binding is first type-checked, then evaluated.

  • Static & Dynamic Environment

    The types come from the static environment, and values from the dynamic environment

  • Immutability
    • When we map a variable x to a value in the dynamic environment, it will always map to that value.
    • If you create another binding with x but it will create a different environment.
    • Why is immutability beneficial?

      It makes aliasing irrelevant. You don’t need t worry about the contents of x changing later. Hence, for example when assigning y = x we can just make y an alias of x rather than a copy which is more efficient.

  • Function bindings

    Called with arguments and has a body that produces a result. The body is type-checked in the environment where the function was defined ‘argument types → result type’. Functions are values.

  • Let expressions

    Allow local variables inside a function body.

    • Why are they useful?

      They make code more readable

      They allow for helper functions that prevent exponentially increasing numbe of recursive calls. Some computation can only be done once.

Week 3

Key Concepts:

  • Compound Types

    Types that contain other types. Eg Lists, options.

    • Creating your own: (3 kinds)
      1. Each of Types (t contains each of t1, t2...). Eg tuples, records.
      1. One of ((t has one of t1, t2..).
      1. Self-reference (t may refer to itself to describe recursive data structures). Eg list
  • Records

    Each of types with named fields. Order doesn’t matter.

  • Syntactic sugar
    • What are tuples actually?

      “tuples are just syntactic sugarfor records with fields named 1, 2, ...,n.”

  • Datatype bindings

    Creates a new type with values int * int or string or nothing. TwoInts, Str and Pizza are constructors (or tags) that tell which variant of our datatype we have. Constructors are either functions that create values of the type, or values themselves. TwoInts is a function of type int*int → mytype. Pizza is a value of type mytype. Values of datatypes are created by calling constructor functions.

    • How to access datatype values?

      Case expressions

    • What are options, actually?

      A polymorphic datatype with variants NONE and SOME of a’

  • Case expressions

    A more powerful if-else statement.

    • Why is it better?

      You cannot forget cases

  • Type inference

    You do not need to write down types, ML will infer them

  • Tail recursion

    a recursive call is tail recursive when there is nothing more for the caller to do after the callee returns except return the callee’s result.

Week 4

Key Concepts:

  • First-class functions

    By “first-class” we mean that functions can be computed, passed, stored, etc.wherever other values can be computed, passed, stored, etc. As examples, we can pass them to functions, return them from functions, put them in pairs, have them be part of the data a datatype constructor carries, etc.

  • Function closures

    Refers to functions that use variables defined outside of them, which makes first-class functions much more powerful.

    The closure overall is “closed”— it has everything it needs to produce a function result given a function argument.

  • What is ‘functional programming’?

    Mainly:

    • using functions as values
    • mostly avoiding mutation

    But also:

    • style that uses a lot of recursion
    • syntax that is closer to mathematical definitions of functions
    • laziness (see later)

  • Parametric polymorphism

    Functions can take any type (generic types, a’ etc)

    • Why is this useful?

      otherwise functions would have to be defined again for each type, instead they can be used more generally

  • Anonymous functions

    functions without names (fn x ⇒ ...)

    • Where are they useful?

      often used as arguments

    • What can they not do?

      Recursion

  • Map, Filter, Fold
    • Map(f, xs) produces a new list applying f to each element
    • Filter(f, xs) produces a list containing only the elements for which f returns true
    • Fold(f, acc, xs) combines elements by applying f(acc,x) to each element given some ‘initial answer’ acc, iterating over a list where f expresses how to ‘combine’ the elements (eg, sum them)
  • Lexical scope

    The body of a function is evaluated in the environment where the function is defined, not the environment where the function is called.

    • why?

      Separation of concern - we know exactly what a function body’s variables will be bound to at each time it’s called.

  • Currying

    A function takes the first conceptual argument and returns another function that takes the second conceptual argument and so on.

    • Which one is more efficient, tupling or currying?

      Almost never matters, but in ML tupling is slightly faster (in other languages might be otherwise)

  • Partial application

    You can provide only some ‘arguments’ of a curried function and use the resulting closure later.

    If arguments are in wrong order, you can make a new one in different order

  • Mutation in ML

    Only used when “updating the state of something so all users of that state can see a change has occurred.

    In ML, you can used mutation by creating a reference (val x = ref e, get contents with !x)

Week 5

Key Concepts:

  • Statically typed language

    All binding types are determined before any part of the program is run (’at compile time’)

  • The value restriction

    A variable in a val binding can only have a polymorphic type if the expression in the val binding is a value or a variable. Function calls are not variables or values.

  • Equivalence

    A function f is equivalent to a function g (or similarly for other pieces of code) if they produce the same answer and have the same side-effects no matter where they are called in any program with anyarguments.

    Equivalence does not require the same running time, the same use of internal data structures, the same helper functions, etc. All these things are considered “unobservable”, i.e., implementation details that do not affect equivalence.
